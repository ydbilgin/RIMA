### Q1 — Sang Hendrix Realtime Parallax Map Builder Deep Analysis

The **Sang Hendrix Realtime Parallax Map Builder** is a highly regarded plugin for RPG Maker MZ that visualizes and streamlines the traditionally tedious workflow of parallax mapping. Under the hood, its architecture, hot-reload mechanism, and UX loop reveal key patterns:

1. **Underlying Pattern: In-Game Editor (In-Process)**
   * **Mechanism:** Rather than an external editor communicating via Inter-Process Communication (IPC) or sockets, Sang’s tool is a **pure in-game editor overlay**. 
   * RPG Maker MZ runs on a JavaScript engine wrapped in **NW.js** (Node-Webkit), meaning the game runs inside a Chromium browser shell with full access to Node.js APIs (such as the `fs` filesystem module).
   * When launched in Test Play mode (triggered by a command-line flag or editor environment), the plugin loads an editor UI layer written in HTML/CSS/JS directly on top of the main **PixiJS** game canvas. Because the editor and game share the same memory space and JavaScript thread, the editor directly mutates the live scene graph in-process. 

2. **Hot-Reload Mechanism**
   * When a designer drags a prop or paints a tile, the editor instantly calls internal PixiJS methods to update the sprite’s transformation matrix (`x`, `y`, `scale`, `rotation`) and sorting parameters.
   * To persist these changes, the plugin leverages Node’s `fs.writeFileSync()` to write the modified layout directly to a custom JSON structure within the project's `/data/` or a dedicated plugin folder. 
   * Because it is in-process, there is no need for a network socket, named pipe, or external file watcher to reflect active edits; the visual game object is modified directly in memory and re-rendered in the next frame.

3. **Authoring Loop UX**
   * **Grid-Free Placement:** Bypasses RPG Maker's default $48\times48$ tile grid, allowing floating, sub-pixel asset positioning.
   * **Real-time Parallax Scrolling:** Designers can adjust depth factors (`factor.x`, `factor.y`) per layer and preview the camera scrolling depth immediately in-editor.
   * **Occlusion Zones & Z-Sorting:** Real-time visual manipulation of the sprite’s Z-index lets designers instantly test whether the player avatar correctly walks behind tall trees/pillars or on top of floor rugs.
   * **Visual Collision Painting:** Rather than typing coordinates, designers use a brush to paint collision vectors directly over the high-fidelity parallax artwork.

4. **RPG Maker MZ Mismatches That Do Not Transfer to Unity**
   * **C# Static Typing & Compilation vs. Dynamic JS:** In RPG Maker, JavaScript objects are dynamically mutable. You can add elements to scene container arrays or patch methods at runtime without recompilation. In Unity, changes to C# scripts in Play Mode reset upon exiting play mode due to domain reloading, unless complex custom serialization structures are written.
   * **Out-of-the-Box Node APIs in Standalone Player:** NW.js provides deep local file access by default even in the player build. In Unity, standalone builds (`.exe`) have access to `System.IO` but completely lack Editor-only reflection capabilities, the `AssetDatabase` API, and scene serialization modules unless explicitly packaged and implemented at runtime.

### Q3 — Asset Layer Architecture in Indie 2D Top-Down ARPGs

#### Layering Conventions in Benchmarked ARPGs

1. **Hades (Supergiant Games):**
   * *Floor & Decals:* Floors are non-sorted textures. Decals (bloodstains, floor cracks, cyan rune circles) are separate sprites placed with small negative sorting offsets relative to characters.
   * *Cliff / Boundary:* Floating island edges. Cliffs are purely visual sprites placed on a lower rendering layer so the floor tiles mask their top seams. Movement boundaries are defined by manually placed 2D polygon colliders.
   * *Walkable Decor:* Debris, flat rune circles, floor chains. `colliderType=None`.
   * *Wall / Blocker:* Heavy stone pillars, braziers, outer boundary walls. These utilize **Circle Collider 2D** or **Capsule Collider 2D** at the base (pivot point) to allow smooth sliding, preventing the player from snagging on corners.
   * *Sorting Order:* Dynamic Y-sorting based on the base pivot of the actor/obstacle sprite.

2. **Children of Morta (Dead Mage):**
   * *Floor & Decals:* Layered tilemap grids. Decals are stamped onto higher tile levels or placed as independent static sprites.
   * *Cliff / Boundary:* Cliff edges utilize 2D tilemaps with sharp, grid-aligned Box/Polygon colliders.
   * *Walkable Decor:* Moss clumps, grass, small bones. No collision logic.
   * *Wall / Blocker:* Rocky outcroppings and ruins. Static colliders are shaped to match the base footing of the sprites.
   * *Sorting Order:* Strict engine-level Y-sorting using the character's feet pivot.

3. **Hyper Light Drifter (Heart Machine):**
   * *Floor:* Flat tilemap layers representing walkable paths.
   * *Cliff / Boundary:* Strong geometric boundaries with rigid grid-based rectangular colliders.
   * *Walkable Decor:* Flat foliage, dirt patches, ground markings.
   * *Wall / Blocker:* Square pillars and walls. Colliders are aligned to the isometric grid.
   * *Sorting Order:* Dynamic Y-sorting combined with explicit Z-index layers for elevated bridges and multi-level platforms.

4. **Death's Door (Acid Nerve):**
   * *Architecture:* Rendered in 3D with an orthographic camera perspective.
   * *Floor/Cliff:* Actual 3D meshes with physical mesh colliders.
   * *Walkable Decor/Blockers:* Columns and interactive elements use cylinder or box colliders at their base to facilitate fluent player traversal.

#### Proposed Asset Layer Architecture for RIMA V1 (Elysium V1)

Accounting for RIMA's lock on a wall-less floating Elysium arena, the South-facing cliff rule (`CliffAutoPlacer`), and the existing asset pack structure, a **6-Layer Split** is highly recommended:

```
[Camera]
   │
   ├── [Layer 5] Gameplay Objects (Dynamic Y-Sort, Order 0)
   ├── [Layer 4] Environment Walls & Obstacles (Dynamic Y-Sort, Order 0)
   │
   ├── [Layer 3] Cliff Face & Mounting Decor (Static/Offset, Order 0)
   ├── [Layer 2] Walkable Floor Decor (Static, Order -1)
   │
   ├── [Layer 1] Cliff Base Layer (Static, Order -2)
   └── [Layer 0] Floor & Decals Layer (Static, Order -10)
```

##### 1. Floor & Decals Layer (Static)
* **Description:** Represents the flat, playable floor grid. Decals like the Cyan Rune circles, dirt cracks, and blood splatters are rendered directly on top.
* **Target Prefabs/Tiles:** `DirectionalCliffTile_Hades.asset` (floor components), Cobblestone/Dirt tiles.
* **Physics:** None.
* **Sorting Layer:** `Floor` | **Sorting Order:** `-10` (Floor) to `-5` (Decals).

##### 2. Cliff Base Layer (Static)
* **Description:** Renders the vertical drop-off cliff faces under the playable deck. In alignment with RIMA's locked system, these face exclusively South, Southeast, and Southwest.
* **Target Prefabs/Tiles:** Generated via `CliffAutoPlacer` on the `CliffTilemap` using `DeterministicVariantTile`.
* **Physics:** None (visual cliff only). Actual cliff-edge boundaries are enforced by 2D polygon boundaries.
* **Sorting Layer:** `Floor` | **Sorting Order:** `-1` (forces the playable floor to cleanly mask the top edge of the cliff drop, achieving the Hades overlap illusion).

##### 3. Cliff Face & Mounting Decor Layer (Static)
* **Description:** Represents decorations attached directly to the vertical cliff faces hanging off the arena edge.
* **Target Prefabs/Tiles:** `mounting_*.prefab` (15 variants including chains, hanging vines, wall-mounted braziers).
* **Physics:** `colliderType=None`.
* **Sorting Layer:** `CliffDecor` | **Sorting Order:** `0` (renders in front of the cliff drop-off base, but behind any gameplay characters standing near the edge).

##### 4. Walkable Floor Decor Layer (Static)
* **Description:** Small props scattered across the playable floor that do not restrict movement.
* **Target Prefabs/Tiles:** Loose bones, stone rubble, flat chains.
* **Physics:** `colliderType=None`.
* **Sorting Layer:** `FloorDecor` | **Sorting Order:** `0` (below characters).

##### 5. Environment Walls & Obstacles Layer (Dynamic Y-Sorted)
* **Description:** Solid architectural hazards and cover elements.
* **Target Prefabs/Tiles:** `StoneColumn`, `statue_*.prefab` (14 variants), brazier stands, pillars.
* **Physics:** **Capsule Collider 2D** or **Circle Collider 2D** placed at the pivot point (feet base). Rigidbodies are configured with `Kinematic` or static constraints.
* **Sorting Layer:** `Default` | **Sorting Order:** Dynamic Y-Sort based on the bottom-center pivot point.

##### 6. Gameplay Objects Layer (Dynamic Y-Sorted)
* **Description:** All active units and interactive elements.
* **Target Prefabs/Tiles:** Player character, 4 Mob types, Loot chests, Map Fragments, gate portals.
* **Physics:** **Capsule Collider 2D** for characters, small trigger zones for items/gates.
* **Sorting Layer:** `Default` | **Sorting Order:** Dynamic Y-Sort.

### Q4 — Indie Game Live Editing References Beyond Sang

* **Hades Modding (SGGModUtil):** Uses a custom bootstrap script to hook into the game’s proprietary Lua runtime. Level layouts and configurations are initialized during load, requiring a **game restart** to register changes.
* **Stardew Valley Content Patcher:** A JSON-driven content management system running via SMAPI. It supports **true runtime reloads** via the `patch reload` console command, allowing textures and data sheets to refresh on-the-fly.
* **Tilesetter Pro:** A standalone tool that relies on a **static export workflow**. Layouts are generated in the application and exported as pre-baked assets to the Unity Editor, lacking any live reload capabilities.
* **LDtk (Level Designer Toolkit):** A modern standalone map editor. It utilizes a **reactive editor-only import loop**; saving the `.ldtk` project file triggers an automatic asset post-processor in Unity that rebuilds the level prefab in the editor.
* **Tiled Map Editor + Super Tiled2Unity:** Implements a similar **reactive editor-only import loop**. Unity detects changes to the `.tmx` file on disk and automatically triggers an asset import rebuild script.

#### Analysis for RIMA
* **Closest Model to Sang's:** **Stardew Valley's Content Patcher** is closest, as it operates a live runtime reload system directly inside an active play session.
* **Best Technical Match for RIMA:** **LDtk / Tiled’s Reactive Editor Loop** is the most practical and stable fit. It completely eliminates Domain Reload friction inside the Unity Editor without requiring the developer to implement and debug a complex, runtime standalone-game IPC bridge.

### Opus Should Know: Critical Synthesis Summary

* **Sang Hendrix’s In-Game Magic is In-Process:** The Realtime Parallax Map Builder works effortlessly because it runs as a JS overlay inside the active NW.js Chromium game thread, directly manipulating the PixiJS canvas scene graph. It does not require sockets or file watchers for its live preview.
* **The "Hades" Cliff Illusion is All About Underlaps:** To perfect RIMA's Elysium aesthetic, the `CliffTilemap` must render on the `Floor` layer with a negative order (`-1`) while using a Y-offset of `1.21875` (exactly 2 cells). This drops the face of the cliff into the void while the playable floor overlaps and covers its top seams, eliminating visual tiling issues.
* **Unity standalone IPC is the sweet spot:** If the user demands a separate game window and editing window, the **Editor Extension + Standalone Player Bridge** (Pattern 2) is the best option. It provides high velocity, runs the game fullscreen separately, and avoids building a custom in-game UI editor from scratch.
