## Slot model (confirmed)
- Slots 0-3 = PRIMARY band (Q/E/R/F). Slots 4-5 = SECONDARY band (Z/X, boss-unlock gated).
- `Warblade_SkillController.SlotCount` = 4 (secondary locked) or 6 (unlocked) — TOTAL, not primary-only.
- Non-Warblade primaries (Elementalist/Ranger/Shadowblade/Ronin) own only 4 slots on their own
  controller; their SECONDARY band lives on the Warblade controller (`AssignActive` host split at slot<4).
- `currentActiveSkills` holds ALL owned skills (primary + secondary combined).
- Secondary-class cards are only offered in the NewUnlock scenario, which fires AFTER
  `UnlockSecondarySlots()` (PlayerClassManager:92) — so a secondary pick always sees an unlocked band.

## BUG 1 — count/cap mismatch (HandleActivePick, was ~490-503) — REAL, FIXED
Old: `if (currentActiveSkills.Count >= maxSlots)` compared the COMBINED owned count against a single
`SlotCount` cap. The new pick lands in a specific band (primary OR secondary), so a half-empty primary
band could flip into replace-mode once the secondary band filled the count to the cap.
- Proof (data, reflection): secondary LOCKED, primary 0,1 filled / 2,3 EMPTY, array slots 4,5 occupied,
  count=4, maxSlots=4. OLD check `4>=4` => replaceTrigger=TRUE (wrongly blocks a 3rd PRIMARY pick).
  NEW `HasFreePrimarySlot()`=TRUE (proceeds into slot 2); `HasFreeSecondarySlot()`=FALSE (locked band →
  a secondary pick correctly routes to replace-mode instead of writing a locked slot). PASS.
- Fix: decide fullness against the TARGETED band via two new probes `HasFreePrimarySlot()` /
  `HasFreeSecondarySlot()` that mirror `FindNext*Slot` host resolution and respect `SecondaryUnlocked`.

## BUG 2 — slot-0 clobber (OnReplaceChosen, was ~509-514) — REAL, FIXED
Old: `if (replaceSlot < 0) replaceSlot = FindNextPrimarySlot();`. Replace-mode only runs when slots are
full, so `FindNextPrimarySlot()` returns its fallback 0 → a failed `FindSlotOf` silently overwrote slot 0.
- Proof (data): all primary slots full, `toReplace` not bound to any slot → `FindSlotOf`=-1. After
  `OnReplaceChosen`: slot0 stays SLOT0_ORIGINAL (NOT the incoming skill), pendingSkill cleared,
  IsDraftActive=false. PASS — abort-safe.
- Fix: on `FindSlotOf < 0`, log + abort the replace (clear pendingSkill, Hide UI, IsDraftActive=false)
  instead of clobbering slot 0. Normal replace flow unchanged.
- Regression proof: `FindSlotOf` succeeds (=2) → slot 2 overwritten with new ChainLightning skill,
  slot 0 untouched. PASS.
