# 1. Branch E camera tilt feasibility
Test plan: duplicate `Assets/Scenes/Demo/_FazMVP_Demo.unity`, capture baseline, then set Main Camera X rotation to 4, 6, 8, and 10 degrees while keeping orthographic, PixelPerfectCamera enabled, assetsPPU=64, ref=480x270, upscaleRT=1. For each angle capture still and short pan: floor grid, Warblade/one mob sprite, L3 wall overlay, Light2D, and sorting overlap. Findings from shell/code inspection: Unity PixelPerfectCamera 5.1.1 does not forbid rotated cameras; it snaps camera position through `worldToCameraMatrix` and corrects orthographic size, so the component will run. The risk is visual, not API failure: rotation makes SpriteRenderer/tilemap quads no longer axis-aligned in camera space, so pixel-perfect sampling can shimmer or stair-step even with point upscale. SpriteRenderer does not auto-billboard; camera tilt will foreshorten characters unless character sprites are inverse-pitched toward the camera or rendered by a separate un-tilted overlay camera. URP 17.3 Renderer2D and Light2D are compatible in the sense that Light2D targets sorting layers and the Renderer2D asset is present with depth stencil enabled, but tilt can expose sorting/depth assumptions because the scene was authored as XY-plane 2D. Safe range: 0-6 degrees likely safest, 8 degrees acceptable only if A/B passes, 10 degrees upper experimental, 12 degrees too risky for MVP without billboard/overlay sprite compensation.

# 2. Branch D floor de-emphasis implementation
`RoomTemplateSO` currently stores tint in `backgroundLayers`; the field is `BackgroundLayerData.tint`, exposed in `RoomTemplateSOInspector`, and applied at runtime by `RoomBankRuntimeTester` through `SpriteRenderer.color = layer.tint`. This makes painted background layer de-emphasis immediately feasible with no schema-risk beyond adding a multiplier or precomputing tinted colors. `MapLayerOrchestrator` currently paints tilemaps through `CornerWangPainter.Paint(...)` and does not read `RoomTemplateSO`, so it cannot apply a RoomTemplateSO field today without a new handoff. For tilemap floors, the light implementation path is: add `floorContrastMultiplier`/`floorSaturationMultiplier` either to `BackgroundLayerData` for painted layers and to the biome/room paint profile for tilemaps, then apply it to `floorTilemap.color` after `CornerWangPainter.Paint` or to each spawned painted layer color before assigning `SpriteRenderer.color`. Existing adjacent precedent: `BiomeSkinApplier` already applies `rule.tint * skin.globalTint` to renderers/tilemaps. Visual test created by shell: `STAGING/tile_angle_review_wang16_saturation_ab.png`, using `STAGING/pixellab_flat_test/wang16_test.png` with a -20% saturation override. Result: saturation reduction helps mood cohesion but does not solve geometry by itself; the blue floor still reads as a square 32x32 tile field, so contrast must also be reduced and L3/L4/L5 must carry the focal weight.
